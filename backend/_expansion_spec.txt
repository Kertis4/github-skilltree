HARD VALIDATION RULES (the loader rejects the file otherwise)
-------------------------------------------------------------
* id: unique, lowercase-kebab, stable (no duplicates with existing or new ids).
* kind: exactly one of: hard | concept
        (hard = a concrete technology/tool; concept = a paradigm/skill)
* domain: exactly one of these 8 ids:
        languages | paradigms | frameworks | architecture | build-tooling |
        data | quality | infrastructure
* detection.evidence: one of: deterministic | llm | hybrid
        - deterministic: resolved from files globs and/or signals (no LLM)
        - llm: needs source-code reading; MUST set `languages` (relevance gate)
        - hybrid: both
* graph.tier: integer 0..5  (0 = foundational; set ~ 1 + max(tier of requires))
* graph.requires: list of EXISTING skill ids (existing or ones added earlier in
        this same file). Must NOT include the skill's own id. The whole graph
        MUST stay ACYCLIC. Prefer 0-2 prerequisites.
* tracks: a map {trackId: weight}. trackId one of:
        frontend | fullstack | backend | infrastructure | systems | mobile
        weight: float 0.0..1.0 (honest affinity; primary track >= 0.6).
* resources: list of {title, url, kind, level}
        kind: docs | tutorial | course | book | video | reference
        level: basic | intermediate | advanced
        Use REAL, well-known, stable URLs (official docs/MDN/etc). 1-2 each.

TRACK-MEMBERSHIP / WEIGHTING GUIDANCE
-------------------------------------
* A skill may belong to MANY tracks. fullstack ~= union of frontend + backend:
  most frontend and backend skills should also carry a fullstack weight ~0.5-0.8.
* Shared foundations (languages, core CS concepts, git, testing) carry a weight
  on most/all tracks (0.4-0.7).
* To "count" toward a track's ~50 at the validator's default threshold, a skill
  needs affinity >= 0.5 for that track. Do not inflate weights dishonestly —
  add genuinely relevant skills instead.
